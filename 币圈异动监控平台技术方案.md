# 币圈异动监控平台技术方案

## 1. 方案目标

本方案用于指导 **CryptoWatchtower** 从 0 到 1 完成 MVP 建设，并为后续产品化与商业化预留清晰演进路径。

平台的核心目标不是做交易执行，而是建设一套稳定、低延迟、可扩展的 **币圈市场异动监控与告警基础设施**，实现以下能力：

- 实时接入 Binance 市场数据
- 识别大单、爆仓、Funding 异常等事件
- 通过 Telegram 向用户和频道分发告警
- 支持规则配置、日志留存、限流防刷屏
- 为后续接入 Discord、Dashboard、AI Summary、多交易所扩展提供架构基础

一句话概括：

> 先把实时监控链路跑通并稳定运行，再逐步产品化，而不是一开始做成大而全的 SaaS 平台。

---

## 2. 分阶段实施策略

结合 MVP 文档和当前仓库状态，建议采用两阶段落地。

### 2.1 Phase 1：核心监控链路 MVP

目标：尽快让真实用户收到稳定的实时提醒。

交付范围：

- Binance `aggTrade` 数据采集
- Binance `forceOrder/liquidation` 数据采集
- Funding Rate 定时抓取
- `MarketEvent` 标准化模型
- Rule Engine 异动判断
- Telegram Bot 告警推送
- PostgreSQL 事件与通知日志存储
- Redis 限流、去重、短期状态缓存
- 基础 HTTP API
- Docker 化部署
- 健康检查、日志、错误恢复

Phase 1 的交付标准不是“功能很多”，而是：

- 数据链路持续稳定
- 告警内容可信
- 推送频率可控
- 服务可连续运行 24 小时以上

### 2.2 Phase 2：产品化与商业化扩展

目标：在 MVP 稳定后，增强对外可用性与转化能力。

交付范围：

- 免费频道推送模式 / 私聊推送模式
- 多 symbol 扩展配置
- Discord Webhook
- 简单 Dashboard
- Landing Page / 等待名单
- 用户自定义规则
- AI 市场摘要原型
- Free / Pro / VIP 权限边界
- 多交易所扩展准备

设计原则：

> Phase 1 先解决“能跑、能用、能稳定提醒”，Phase 2 再解决“能增长、能转化、能收费”。

---

## 3. 架构设计原则

本项目推荐采用：

> 模块化单体（Modular Monolith） + 容器化部署 + 状态外置存储

原因是 MVP 阶段最重要的是开发速度、运行稳定性和问题可定位性，而不是微服务拆分的形式感。

### 3.1 为什么不建议一开始微服务化

当前业务规模下，微服务会带来额外复杂度：

- 服务间通信和序列化成本
- 本地联调复杂
- 部署链路更长
- 日志、追踪、告警成本更高
- 早期边界未稳定，过早拆分容易返工

因此首版应采用一个 Go 主服务，内部按职责拆分模块：

- `collector`
- `eventbus`
- `rule`
- `notifier`
- `storage`
- `api`
- `scheduler`

### 3.2 为什么仍要保留未来可拆分边界

虽然首版不拆服务，但模块边界必须提前设计好。后续非常可能出现这些演进需求：

- 多交易所 collector 独立扩展
- Rule Engine 独立横向扩容
- Notification 独立处理高并发推送
- API 层独立服务化
- AI Summary 异步任务化

所以代码层面必须遵守这些原则：

- 模块间通过清晰接口协作
- 内部事件结构统一
- 配置中心化
- 存储访问通过 repository 或 storage 层隔离
- 不让 Telegram、Binance SDK、数据库细节渗透到全局

这意味着：

> 部署上先单体，设计上按未来可拆分来约束。

## 4. 总体架构

平台总体链路建议如下：

```text
Binance WS / REST
   ↓
Collector（采集与标准化）
   ↓
In-Process Event Bus
   ↓
Rule Engine（规则判断）
   ↓
Alert Pipeline（去重 / 限流 / 格式化）
   ↓
Notifier（Telegram）
   ↓
用户 / 群组 / 频道

同时：
- PostgreSQL 持久化事件、规则、用户、通知日志
- Redis 存储限流键、去重键、短时聚合状态
- HTTP API 提供健康检查、规则查看、测试告警等能力
- Scheduler 负责 Funding 拉取和后续摘要任务
```

这个架构的关键点有两个。

### 4.1 数据链路以“事件”为中心

所有上游输入，不管是成交、爆仓还是 Funding，都要统一进入 `MarketEvent` 或其衍生结构。Rule Engine 不直接处理 Binance 原始 JSON，而只处理标准化后的内部事件。

这样做的好处：

- 降低上游交易所差异影响
- 方便后续接入 OKX / Bybit
- 便于测试和回放
- 便于把异常事件持久化

### 4.2 告警链路以“规则命中结果”为中心

Rule Engine 输出的不应是“直接发 Telegram”，而应是统一的 `Alert` 对象。是否发送、发给谁、是否限流、如何格式化，属于下游 Notification 责任。

这样做能避免后期出现：

- 规则逻辑和推送逻辑耦合
- 多渠道通知难以扩展
- 不同用户等级的分发策略难以接入

## 5. 默认部署方案

本方案默认按以下目标部署：

> 云服务器 + 容器化部署，保留未来迁移 Kubernetes 的边界

### 5.1 推荐部署形态

首版生产建议：

- 1 台应用云主机
- Docker Compose 编排
- Nginx 或 Caddy 作为反向代理
- PostgreSQL 与 Redis 初期可同机部署
- 后续可平滑迁移到托管数据库

推荐原因：

- 成本低
- 部署速度快
- 适合 MVP 迭代
- 故障排查路径短
- 与本地开发环境一致性高

### 5.2 云平台性价比推荐

#### 首选：Hetzner Cloud

推荐理由：

- 同价位 CPU / 内存资源通常比主流云更划算
- 对 Go 后端、Redis、PostgreSQL 这类轻中型负载非常友好
- 很适合早期 MVP、Side Project、小团队 SaaS 冷启动
- 可先用较低成本机器验证产品和监控链路

适用场景：

- 早期单机部署
- 日告警量中低规模
- 团队希望优先压低基础设施成本

#### 备选：DigitalOcean

推荐理由：

- 控制台、文档、社区教程成熟
- 部署体验更开箱即用
- 更适合希望少折腾基础设施的团队

缺点：

- 同配置下通常价格高于 Hetzner

#### 亚洲低延迟备选：Vultr Tokyo / Singapore

如果未来主要用户在亚洲，且对 Telegram 推送链路延迟更敏感，可以考虑：

- `Vultr Tokyo`
- `Vultr Singapore`

它们的优势不一定是绝对最低价，而是区域网络路径通常更友好。

### 5.3 首版推荐机器配置

建议从以下规格起步：

- `2 vCPU`
- `4 GB RAM`
- `80 GB SSD`

承载内容：

- Go 主服务
- PostgreSQL
- Redis
- 反向代理
- 基础监控与日志

如果后续出现以下情况，再升级配置或拆分部署：

- 监控币种数量显著增长
- 告警推送量明显上升
- 引入 Dashboard、AI Summary、Discord 等额外模块
- 需要更高的高可用能力

## 6. 技术选型建议

### 6.1 后端语言

- `Go 1.22+`

原因：

- 并发模型适合实时采集与通知分发
- 内存和部署成本相对可控
- 单二进制部署体验好
- 与 Binance WS / REST 场景非常契合

### 6.2 数据库

- `PostgreSQL 16`

原因：

- 结构化数据模型清晰
- 支持 JSONB 存储原始事件
- 查询能力比轻量 KV 更适合事件检索和规则管理

### 6.3 缓存 / 限流 / 短期状态

- `Redis 7`

用途：

- 告警限流
- 去重键
- 1 分钟窗口聚合统计
- 后续可演进为 stream 或任务协调

### 6.4 API

- `HTTP REST`
- 推荐使用轻量路由框架，如 `chi` 或 `gin`
- MVP 更推荐 `chi`，更轻、更贴近标准库

### 6.5 配置管理

- `YAML + 环境变量覆盖`

### 6.6 容器化

- `Docker + Docker Compose`

### 6.7 日志

- `结构化日志（JSON）`
- 推荐 `zap` 或 Go 标准 `slog`

## 7. 交易所接入优先级与扩展策略

### 7.1 接入优先级原则

交易所接入策略不以“接得越多越好”为目标，而以“最小复杂度下获得最大监控价值”为原则。对当前阶段而言，平台最需要的是：

- 足够活跃的交易深度
- 稳定且文档完善的 WebSocket / REST 接口
- 覆盖主流交易对和衍生品场景
- 能支撑大单、爆仓、Funding 三类规则

因此，首版不建议同时对接多个交易所。多交易所并行接入会显著增加以下复杂度：

- 不同交易所事件模型不一致
- WebSocket 重连与心跳策略不同
- REST 限频规则不同
- 交易对命名规则不同
- 爆仓、Funding、主动买卖方向定义存在差异
- 测试与回放样本成倍增加

从 MVP 落地角度看，同时处理这些差异会明显拖慢首版交付，且会稀释对稳定性的投入。

### 7.2 Phase 1 仅接入 Binance

Phase 1 明确只接入 **Binance**，作为唯一首发交易平台。

原因如下：

- Binance 现货与合约市场活跃度高，足以覆盖 MVP 验证所需的大部分市场异动场景
- 官方 API 文档、SDK 生态和社区样例相对成熟
- `aggTrade`、`forceOrder`、Funding Rate 等接口已经能满足首版核心规则
- 对主流币种如 `BTCUSDT`、`ETHUSDT`、`SOLUSDT` 的监控价值足够高
- 用单一交易所先打磨“采集 -> 标准化 -> 规则 -> 告警”全链路，更容易定位问题与提升可靠性

结论是：

> 首版先只做 Binance，目标是降低接入复杂度、压缩交付周期、优先打磨监控链路稳定性。

### 7.3 Phase 2 Top 5 扩展清单

在首版稳定后，再按优先级扩展其他中心化交易所。建议的 Top 5 关注名单如下：

1. `Binance`
2. `Bybit`
3. `OKX`
4. `Gate.io`
5. `Bitget`

其中：

- `Binance` 为 Phase 1 首发平台
- `Bybit` 与 `OKX` 作为 Phase 2 第一优先扩展对象
- `Gate.io` 与 `Bitget` 作为第二批扩展对象

这个排序主要基于以下维度综合判断：

- 衍生品市场活跃度
- 主流币种覆盖度
- API 接入成熟度
- 异动监控样本丰富度
- 国际用户认知度

### 7.4 扩展设计要求

虽然 Phase 1 只接入 Binance，但技术方案必须从第一天开始为多交易所扩展保留边界。建议采用“交易所适配层”模式。

推荐抽象：

```go
type ExchangeCollector interface {
    Name() string
    Start(ctx context.Context) error
    Subscribe(symbols []string) error
}
```

标准化输出统一进入内部事件模型，例如：

```go
type MarketEvent struct {
    ID         string
    Exchange   string
    Symbol     string
    EventType  string
    Side       string
    Price      float64
    Quantity   float64
    Notional   float64
    RawPayload []byte
    EventTime  time.Time
    CreatedAt  time.Time
}
```

要求如下：

- 每个交易所只负责“采集与标准化”，不直接参与规则判断
- Rule Engine 只消费统一事件，不关心事件来自哪个交易所
- Notifier 不依赖交易所原始字段
- Symbol 映射、事件类型映射、字段兼容逻辑集中放在 adapter / normalizer 层

这样后续增加 `BybitCollector`、`OKXCollector` 时，只需要扩展采集适配和标准化逻辑，而不必重写规则引擎和通知模块。

### 7.5 多交易所扩展节奏建议

建议的扩展顺序如下：

#### 第一步：单交易所稳定化

先让 Binance 版本满足以下标准：

- 连续运行 24 小时以上
- 自动重连可靠
- 事件入库完整
- 告警误报率可接受
- Telegram 限流策略稳定

#### 第二步：抽象交易所接口

在 Binance 版本验证稳定后，重构并固化以下边界：

- `collector` 接口化
- `symbol` 规范统一
- `event_type` 枚举统一
- `funding` 与 `liquidation` 事件语义标准统一

#### 第三步：接入 Bybit 与 OKX

优先选择与 Binance 规则模型相似、市场活跃度高的平台，以最低心智成本完成第二、第三个交易所扩展。

#### 第四步：接入 Gate.io 与 Bitget

补充长尾交易对和更多衍生品异动样本，提升监控广度。

该节奏的目的不是追求“平台数量”，而是追求：

> 每新增一个交易所，都只增加有限复杂度，并显著增加可监控市场覆盖面。

## 8. 模块设计

为保证 MVP 阶段开发效率与后续可扩展性，系统采用模块化单体结构。所有模块运行在同一个 Go 进程中，但通过清晰接口和职责边界解耦。

推荐目录结构如下：

```text
crypto-watchtower/
├── cmd/
│   └── server/
│       └── main.go
├── internal/
│   ├── config/
│   ├── collector/
│   ├── eventbus/
│   ├── rule/
│   ├── notifier/
│   ├── storage/
│   ├── api/
│   ├── model/
│   └── scheduler/
├── migrations/
├── deployments/
├── configs/
├── scripts/
└── docs/
```

### 8.1 `config` 模块

职责：

- 读取 YAML 配置文件
- 支持环境变量覆盖
- 统一管理应用级配置项
- 为所有模块提供强类型配置对象

建议配置分类：

- 应用基础配置
- Binance 接入配置
- PostgreSQL 配置
- Redis 配置
- Telegram 配置
- HTTP 服务配置
- 规则默认阈值配置
- 调度任务配置

要求：

- 配置在启动时一次性完成校验
- 缺失关键配置时直接启动失败
- 不允许业务模块自行读取环境变量

### 8.2 `collector` 模块

职责：

- 建立 Binance WebSocket 连接
- 调用 Funding REST API
- 自动重连、心跳检测、异常恢复
- 将外部事件标准化为内部统一事件
- 将事件投递到 `eventbus`

建议拆分为：

- `binance_ws.go`：处理 `aggTrade` 与 `forceOrder` 订阅
- `binance_rest.go`：处理 Funding Rate 拉取
- `normalizer.go`：将 Binance 原始消息转换为 `MarketEvent`

设计原则：

- 采集模块只负责“拿数据”和“标准化”
- 不在采集模块内做业务规则判断
- 不在采集模块直接发送通知

### 8.3 `eventbus` 模块

职责：

- 在模块间传递标准化事件
- 对上游采集与下游规则引擎做运行时解耦
- 为后续切换 Redis Stream 或消息队列保留接口

MVP 建议采用：

- 进程内 channel 实现

推荐接口：

```go
type EventBus interface {
    Publish(ctx context.Context, event model.MarketEvent) error
    Subscribe(ctx context.Context) <-chan model.MarketEvent
}
```

说明：

- Phase 1 采用 in-process bus，简化复杂度
- Phase 2 若出现多实例扩展需求，再引入 Redis Stream 或 Kafka

### 8.4 `rule` 模块

职责：

- 消费 `MarketEvent`
- 执行告警规则判断
- 完成基础去重与限流判定
- 输出统一 `Alert`

Phase 1 建议先实现三类规则：

- 大单异动规则
- 爆仓规则
- Funding 异常规则

内部建议结构：

- `engine.go`
- `large_trade.go`
- `liquidation.go`
- `funding.go`

约束：

- 规则逻辑必须纯粹，不依赖 Telegram 或 HTTP
- 单条规则应可被独立测试
- 支持通过配置或数据库加载阈值

### 8.5 `notifier` 模块

职责：

- 消费 `Alert`
- 执行消息格式化
- 调用 Telegram Bot API
- 记录通知日志
- 失败重试
- 预留 Discord / Webhook 扩展能力

推荐接口：

```go
type Notifier interface {
    Send(ctx context.Context, alert model.Alert) error
}
```

推荐拆分：

- `telegram.go`
- `formatter.go`
- `discord.go`（Phase 2 预留）

要求：

- 统一在该模块处理 Markdown 格式
- 统一在该模块处理渠道级限流
- 不让 Rule Engine 感知消息模板细节

### 8.6 `storage` 模块

职责：

- 管理 PostgreSQL 连接
- 管理 Redis 连接
- 提供 repository 级别的数据访问
- 屏蔽底层 SQL 与缓存细节

建议拆分为：

- `postgres.go`
- `redis.go`
- `market_event_repo.go`
- `alert_rule_repo.go`
- `notification_log_repo.go`
- `user_repo.go`

要求：

- SQL 访问集中管理
- 写入失败必须有明确错误返回
- 关键写路径需带上下文超时控制

### 8.7 `api` 模块

职责：

- 暴露健康检查接口
- 提供规则管理接口
- 提供测试告警接口
- 提供 Telegram webhook/polling 管理辅助能力

建议拆分：

- `router.go`
- `health.go`
- `rules.go`
- `telegram.go`
- `alerts.go`

原则：

- API 只做入参校验、调用 service、输出响应
- 不将复杂业务逻辑直接写进 handler

### 8.8 `scheduler` 模块

职责：

- 周期性拉取 Funding 数据
- 执行后续 AI Summary 定时任务
- 执行清理、补偿或聚合类后台任务

Phase 1 仅需：

- Funding Rate 定时抓取任务

建议：

- 使用独立 goroutine + ticker 实现
- 所有任务必须支持 `context.Context` 退出控制

## 9. 核心数据模型设计

本系统以“事件”和“告警”为核心业务实体。推荐在 `internal/model` 下集中定义。

### 9.1 MarketEvent

用于表示标准化后的市场事件。

```go
type MarketEvent struct {
    ID         string
    Exchange   string
    Symbol     string
    EventType  string
    Side       string
    Price      float64
    Quantity   float64
    Notional   float64
    Metadata   map[string]any
    RawPayload []byte
    EventTime  time.Time
    CreatedAt  time.Time
}
```

字段说明：

- `Exchange`：交易所标识，Phase 1 固定为 `binance`
- `Symbol`：统一交易对标识，如 `BTCUSDT`

- `Metadata`：扩展字段，用于保存 Funding、额外标签等补充信息

## 11. Rule Engine 设计

Rule Engine 是系统核心。其目标不是“复杂策略分析”，而是快速、稳定、可解释地识别明确市场异常。

### 11.1 规则引擎职责边界

Rule Engine 负责：

- 消费标准化事件
- 加载并应用规则配置
- 完成阈值判断
- 基于 Redis 执行去重与限流
- 输出统一 `Alert`

Rule Engine 不负责：

- 直接发送 Telegram
- 管理 HTTP 请求
- 持有 Binance 原始连接

### 11.2 规则分类

Phase 1 规则分为三类。

#### 1. 大单异动规则

判定条件示例：

- 单笔成交额 `>= 100,000 USDT`
- 或 60 秒窗口内主动买入 / 主动卖出累计 `>= 500,000 USDT`

说明：

- 单笔规则适合快速提醒
- 窗口累计规则适合捕捉短时异常成交堆积

#### 2. 爆仓规则

判定条件示例：

- 单笔爆仓金额 `>= 100,000 USDT`

说明：

- 首版先使用单笔阈值
- 后续可扩展爆仓聚集度规则

#### 3. Funding 异常规则

判定条件示例：

- Funding Rate `>= 0.08%`
- 或 Funding Rate `<= -0.08%`

说明：

- Funding 不是 WS 高频流，适合通过定时任务抓取
- 输出偏风险提示，不输出买卖建议

### 11.3 规则执行流程

推荐执行流程：

```text
事件进入 Rule Engine
   ↓
按 event_type 路由到对应规则处理器
   ↓
加载适用规则
   ↓
执行阈值判断
   ↓
生成 trigger_key
   ↓
检查 Redis 去重 / 限流
   ↓
命中则生成 Alert
   ↓
写入 alerts 表
   ↓
交给 notifier
```

### 11.4 去重与限流设计

建议采用双层控制。

#### 告警级去重

目标：

- 防止同一事件被重复发送

Redis Key 示例：

```text
dedupe:alert:{exchange}:{symbol}:{rule_type}:{event_id}
```

TTL 建议：

- `120 秒`

#### 规则级限流

目标：

- 防止同一 symbol + 规则在短时间连续刷屏

Redis Key 示例：

```text
rate_limit:alert:{exchange}:{symbol}:{rule_type}
```

TTL 建议：

- `60 秒`

#### 渠道级限流

目标：

- 防止单一 chat 或全局 Telegram 请求过载

Redis Key 示例：

```text
rate_limit:telegram:{chat_id}
rate_limit:telegram:global
```

建议策略：

- 同一 `symbol + rule_type`：60 秒内最多推送 1 次
- 同一 `chat_id`：每分钟最多 20 条
- 全局 Telegram：每秒最多 5 次请求

### 11.5 规则配置来源

Phase 1 建议采用“双来源模式”：

- 默认规则来自配置文件
- 用户规则或动态规则来自数据库

优点：

- 冷启动快
- 默认阈值可直接上线
- 后续可平滑引入动态规则管理

## 12. Notification 与 Telegram 设计

Telegram 是 Phase 1 唯一通知渠道，因此需要优先保证稳定性和可控性。

### 12.1 Telegram 模块职责

- 管理 Bot Token
- 处理消息格式化
- 向私聊、群组、频道发送消息
- 记录发送日志
- 处理发送失败与基础重试

### 12.2 消息模板规范

首版建议统一由 `formatter` 模块生成文本，避免在业务逻辑中拼接字符串。

大单提醒模板：

```text
🚨 {{symbol}} 大额{{side}}

价格: {{price}}
成交额: {{notional}} USDT
交易所: {{exchange}}
时间: {{event_time}}

规则: 单笔成交额超过 {{threshold}} USDT
```

爆仓提醒模板：

```text
💥 {{symbol}} 大额爆仓

方向: {{side}}
金额: {{notional}} USDT
价格: {{price}}
时间: {{event_time}}
```

Funding 提醒模板：

```text
⚠️ {{symbol}} Funding 异常

当前 Funding: {{funding_rate}}
阈值: {{threshold}}
交易所: {{exchange}}
```

### 12.3 发送失败处理

建议策略：

- 网络抖动类错误：短暂重试 1~3 次
- 参数错误类错误：直接失败并记录日志
- 目标 chat 无效：记录后标记用户状态待处理

### 12.4 Telegram 接入方式

Phase 1 推荐优先：

- `polling`

原因：

- 部署更简单
- 适合早期 MVP

Phase 2 可切换：

- `webhook`

适合：

- 对接更规范的生产入口
- 有公网 HTTPS 域名时使用

## 14. Web Dashboard 与前端技术栈设计

Web Dashboard 在本项目中不是 Phase 1 的核心交付，但需要在技术方案中提前定义清楚，以避免后续前后端边界混乱。建议将后台能力拆分为两个层次：

- 内部运维后台
- 用户侧 SaaS 后台

总体策略是：

> 先做内部运维后台，服务于系统运行、规则调整和测试验证；后做用户侧 SaaS 后台，服务于订阅用户、自定义规则和告警历史查询。

### 14.1 后台分层定位

#### 1. 内部运维后台

定位：

- 面向项目运营者和开发者
- 用于查看系统状态、检查规则、测试通知链路、观察近期事件

目标：

- 帮助团队快速判断系统是否健康
- 降低仅靠日志排查问题的成本
- 为后续产品化提供基础页面壳

首批页面建议：

- 服务健康状态页
- 最近告警列表页
- 当前 symbols 与交易所配置页
- 当前系统规则页
- Telegram 测试告警页
- 近期市场事件页

#### 2. 用户侧 SaaS 后台

定位：

- 面向最终订阅用户
- 用于管理个人提醒、查看历史告警、理解订阅权益

目标：

- 将 Telegram 机器人能力产品化
- 支撑 Free / Pro / VIP 分层
- 为未来订阅、支付和多用户规则管理提供入口

后续页面建议：

- 登录与账户页
- 订阅与套餐页
- 个人规则管理页
- 告警历史页
- 通知渠道页
- 使用概览页

### 14.2 分阶段建设策略

#### Phase 1：不做完整后台，仅保留 API

当前阶段仍以 Go 后端 API 为主，不要求上线完整前端页面。此阶段后端应优先保证：

- `/health` 可用
- `/api/v1/symbols` 可用
- `/api/v1/rules` 可用
- `/api/v1/alerts/test` 可用
- `/api/v1/telegram/test` 可用

这样可以先通过 curl、Postman、脚本或简单内部页面进行联调。

#### Phase 1.5：上线内部运维后台

建议在核心实时链路稳定后，优先建设一套轻量内部运维后台，而不是直接进入完整用户后台。

交付建议：

- 基础登录保护或固定 Bearer Token 代理保护
- 健康状态面板
- 规则查看与简单修改
- 最近告警列表
- Telegram 测试操作
- 最近事件查询

该阶段的目标不是美观营销，而是：

> 让系统“可观测、可调试、可操作”。

#### Phase 2：建设用户侧 SaaS 后台

当内部后台验证稳定后，再扩展用户侧功能：

- 用户登录
- 权限与订阅分层
- 个人规则配置
- 通知目标管理
- 告警历史查询
- Dashboard 数据概览

### 14.3 推荐前端技术栈

本项目推荐前端技术栈如下：

- `Next.js`
- `React`
- `TypeScript`
- `Tailwind CSS`
- `shadcn/ui`
- `TanStack Query`
- `Recharts` 或 `Apache ECharts`

推荐原因如下。

#### 1. 使用 Next.js

原因：

- 同时适合内部后台和外部 Landing Page
- 支持服务端渲染与静态生成
- 便于后续增加登录、订阅和 SEO 页面
- 生态成熟，适合中小型 SaaS 产品

建议：

- Dashboard 页面采用 App Router
- 营销页与后台页共用一个前端工程

#### 2. 使用 React + TypeScript

原因：

- 与 Next.js 搭配自然
- 有利于管理规则配置、表格、图表等复杂交互
- TypeScript 能约束前后端接口模型，降低字段漂移风险

#### 3. 使用 Tailwind CSS + shadcn/ui

原因：

- 开发效率高
- 适合快速搭建后台页面
- 组件可控，不受重型组件库约束
- 更容易统一风格和设计 token

设计方向建议：

- 内部运维后台以安静、紧凑、信息密度高为主
- 用户侧 SaaS 后台比运维后台更偏产品化，但仍避免花哨营销感

#### 4. 使用 TanStack Query

原因：

- 适合管理 Dashboard 中的远程状态
- 便于轮询健康状态、告警列表、规则列表
- 能减少手写缓存与刷新逻辑

#### 5. 图表库选择

建议：

- 内部运维后台优先 `Recharts`
- 若后续出现更复杂的成交量、Funding、爆仓分布图，再引入 `Apache ECharts`

原因：

- `Recharts` 上手快，适合轻量图表
- `ECharts` 表达力更强，但复杂度更高

### 14.4 前后端接口边界

为避免未来前端接入时返工，建议现在就定义边界：

- Go 服务继续作为唯一后端 API 提供者
- 前端不直接连接交易所，不直接读取 Redis，不直接访问数据库
- 所有页面数据都通过 HTTP API 获取

前端首批需要消费的接口建议包括：

```text
GET  /health
GET  /api/v1/symbols
GET  /api/v1/rules
POST /api/v1/rules
GET  /api/v1/events
GET  /api/v1/alerts
POST /api/v1/alerts/test
POST /api/v1/telegram/test
```

其中：

- `/api/v1/events`
- `/api/v1/alerts`

可作为内部运维后台新增接口优先实现。

### 14.5 内部运维后台页面设计建议

### 14.6 用户侧 SaaS 后台页面设计建议

后续用户侧后台建议以“个人监控台”为中心，而不是传统复杂管理后台。

建议页面包括：

- 我的规则
- 我的告警历史
- 我的通知渠道
- 我的订阅计划
- 市场摘要

其中：

- Free 用户重点看频道引导与升级入口
- Pro 用户重点看实时提醒、自定义规则和私聊推送
- VIP 用户重点看更多币种、Webhook、AI Summary 与更高频率能力

### 14.7 部署建议

前端建议独立部署，与 Go API 分离：

- 前端：`Vercel` 或 `Cloudflare Pages`
- 后端 API：云主机 Docker Compose

原因：

- 前端迭代频率通常高于后端
- 独立部署有利于后续上线 Landing Page 与 Dashboard
- CDN 分发更适合前端静态资源

若早期追求极简，也可以：

- 前端单独构建
- 由 Go 服务反向代理静态资源

但该方案只建议在早期临时使用，不建议作为长期结构。

### 14.8 最终建议

对于本项目，前端建设的推荐顺序是：

1. Phase 1：仅 API，不做完整后台
2. Phase 1.5：做内部运维后台
3. Phase 2：做用户侧 SaaS 后台

前端技术栈推荐统一为：

> Next.js + React + TypeScript + Tailwind CSS + shadcn/ui + TanStack Query

这样既能满足内部运维后台的快速建设，也能支撑后续 Landing Page、订阅页和用户侧 Dashboard 的持续扩展。

## 16. 稳定性、监控与容灾设计

对于该类实时监控系统，稳定性优先级高于功能数量。

### 16.1 WebSocket 稳定性

必须实现：

- 自动重连
- 心跳检测
- 异常关闭日志记录
- 重连退避策略

建议：

- 初始重连间隔 3~5 秒
- 连续失败时做指数退避，上限 60 秒

### 16.2 幂等与重复控制

必须防止：

- 同一市场事件重复入库
- 同一告警重复推送

方式：

- 事件唯一 ID
- Redis 去重键
- 数据库唯一索引

### 16.3 日志策略

日志必须结构化，并至少覆盖以下维度：

- collector 连接建立与断开
- 收到事件数
- 规则命中数
- 通知发送成功数 / 失败数
- API 请求日志
- 调度任务执行日志

### 16.4 监控指标建议

建议后续接入 Prometheus 指标，至少包括：

- WS 连接状态
- 每分钟事件吞吐
- 每分钟告警生成数
- Telegram 发送成功率
- PostgreSQL 写入耗时
- Redis 操作耗时

### 16.5 容灾策略

Phase 1 不追求高可用集群，但至少要做到：

- 容器崩溃自动重启
- 进程 panic 后自动退出并由容器拉起
- 数据库定期备份
- 配置文件与密钥独立管理

## 17. 安全与合规边界

本平台属于市场监控与风险提醒工具，不属于交易执行或投资顾问系统。

必须坚持以下边界：

- 不承诺收益
- 不提供投资建议
- 不做自动交易
- 不托管用户资金
- 不接触用户私钥
- 不使用“稳赚”“内幕”“高胜率”等营销词

建议在 README、Landing Page、Telegram Bot 帮助信息中统一加入声明：

```text
CryptoWatchtower only provides real-time market telemetry and alerting.
It is not financial advice. Users are responsible for their own decisions.
```

中文版本：

```text
CryptoWatchtower 仅提供实时市场数据监控和风险提醒，不构成任何投资建议。用户需自行承担交易决策风险。
```

测试设计应围绕“规则正确性”和“链路稳定性”展开。

## 22. 最终建议

本项目最重要的不是一次性做出很多功能，而是尽快把如下主链路做扎实：

> Binance 实时采集 -> 标准化 -> 规则判断 -> 限流去重 -> Telegram 推送

在此基础上，再逐步扩展产品能力和商业能力。对于当前阶段，最优策略是：

- 技术上保持模块化单体
- 业务上只做 Binance 首发
- 部署上选择单机云主机容器化
- 产品上优先让真实用户尽快看到高质量实时提醒

这条路线可以在控制复杂度的前提下，最快验证产品价值，并为后续多交易所、多用户和商业化扩展打下足够稳的基础。
